# Review: FE-047 (GradeDetailPage) + FE-048 (SchoolViewPage) — Page Test Suites

## Verdict: APPROVE

## Summary

48 new tests across the School academic-structure overview (Level 1) +
grade detail (Level 2). 1184/1184 full-suite green. Both pages exercise
TanStack Query, Routes/Route param plumbing, modal flows, and CRUD
mutations against a fully-mocked `academicsService`.

## Critical Issues

None.

## Major Issues

None.

## Minor Issues

### 1. `GradeDetailPage` mutation tests assert mock-call, not UI update

Example: lines 401-407 assert `svc.createSection.toHaveBeenCalledWith(...)`
but don't subsequently verify a new card appears. This is a defensible
boundary — the test verifies the contract sent to the service, and the
service-mocking pattern means there's no in-memory cache that would
auto-render the new row. Adding a follow-up `findByText('Section C')`
after configuring the mock to return the new row would strengthen the
assertion to user-observable behavior. Non-blocking.

### 2. `SchoolViewPage` retry assertion uses `toHaveBeenCalledTimes(2)`

Verifies the click triggers a re-fetch but not that fresh data renders.
Same shape as item 1 — the service mock would need to flip between
error and success on subsequent calls to fully prove recovery.
Non-blocking.

### 3. `getByText('--')` for unassigned teacher

`GradeDetailPage.test.tsx` line ~285. `--` is a generic dash; if the
component ever uses it elsewhere the matcher could collide. Scoping
the query inside the section card row would be safer. Style nit.

### 4. Some duplicated-text checks use `getAllByText(...).length >= 1`

`SchoolViewPage.test.tsx` lines ~213-222. Pragmatic given the
"1 grade" / "2 grades" phrasing reappears in sub-labels and cards;
documented intent in the test names is clear.

## Positive Observations

- **`Routes`/`Route` with `MemoryRouter`** to supply `:gradeId` /
  `:sectionId` — exercises the real param-extraction path rather than
  stubbing `useParams`.
- **TanStack Query with `gcTime: 0` + `retry: false`** — clean
  per-test isolation, no inter-test state leakage.
- **Confirm-dialog flow tested** for delete-section (line ~530s) —
  covers a real interaction sequence (open dropdown → click Delete →
  confirm) that's easy to skip.
- **Grade-not-found test** (when `:gradeId` doesn't match any band)
  is the kind of edge case that's typically missed.
- **Empty-state CTAs tested** — "Configure Academic Structure" navigates
  to settings; "Create First Section" opens the modal.
- **Error-state retry button** is exercised on both pages.

## Recommendations (non-blocking)

- For mutation tests: configure the mock to return the new row, then
  assert it appears in the DOM as a follow-up — turns "we sent the
  right call" into "the user sees the result."
- Scope the `'--'` query into a card-row container.
