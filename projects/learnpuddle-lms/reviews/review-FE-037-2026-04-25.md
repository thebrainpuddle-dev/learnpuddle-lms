## Minor Issues

1. **`expect(screen.getByText(/1/))` for the selected-count badge** (line 357).

   ```ts
   await waitFor(() =>
     expect(screen.getByText(/1/)).toBeInTheDocument(),
   );
   ```

   Regex `/1/` will match any DOM text containing the digit "1" â€” `EMP001`, `2026-01-01`, `t-1`, etc. The fixtures expose multiple such strings. The test passes today because either (a) the BulkActionsBar count is the most recently-rendered "1" and `getByText` returns first match, or (b) RTL throws on multiple matches and the test is flaky. Either way, brittle. Tighten with `screen.getByText(/^1$/)` or `within(bulkActionsBar).getByText('1')`. Non-blocking; flag for a sweep.

2. **`screen.getByText('2')` for the bulk-select count** (line 372) â€” same issue as #1, tighter today because no fixture contains `'2'` as standalone text, but worth applying the same `/^2$/` discipline for consistency.

3. **Dual-layout assertion `expect(deactivateBtns.length).toBe(1)`** (line 195) is correct today but tightly coupled to the "active vs inactive teacher render asymmetry":

   - `TEACHER_ALICE` (active) â†’ 1 mobile-card Deactivate button
   - `TEACHER_INACTIVE` (Carol) â†’ 0 mobile-card Deactivate button
   - Total = 1 âś…

   If the desktop table ever gains a textual "Deactivate" affordance (instead of icon-only), this assertion will start counting two â€” one desktop + one mobile per active teacher. Since jsdom doesn't apply Tailwind's responsive classes, both layouts render simultaneously. The risk is real but contingent. Suggested forward-fix: anchor on `within(mobileCardContainer)`. Defer.

4. **Edit modal "Cancel without calling updateTeacher" test does not cover "Cancel after typing"** (line 290â€“301).

   The test opens the edit modal and immediately clicks Cancel. Pressing Cancel after typing into a field â€” the more interesting case for a "no API call on cancel" guarantee â€” isn't exercised. Cosmetic; the basic path is covered.

5. **`mockNavigate.mockReset()`** is called in `beforeEach` (line 144) AND `vi.resetAllMocks()` runs immediately before. The latter resets all `vi.fn()` instances including `mockNavigate` (which is itself a `vi.fn()`). The explicit reset is redundant but harmless. Style nit.

6. **Default `svc.updateTeacher` mock returns `{ ...TEACHER_ALICE, first_name: 'Alicia' }`** (line 148) â€” the expected post-update shape. The "submitting calls updateTeacher with updated data" test (line 251) types `'Alicia'`. If the test ever asserted on the *response* effect (e.g. an updated row in the list), the mock would lock the assertion to one specific name. Right now the mock's resolved value is unused, so this is fine; just noting.

## Positive Observations

- **`within(dialog)` to disambiguate the two "Deactivate" buttons** (line 329-331). After clicking the mobile-card "Deactivate" button, both the card button AND the Headless UI Dialog confirm button (which inherits the `confirmLabel="Deactivate"` prop) live in the DOM. Targeting via `within(await screen.findByRole('dialog')).getByRole('button', { name: /^deactivate$/i })` resolves cleanly without coupling to internal IDs. This is exactly the pattern that should be in the team's testing playbook.
- **`findByRole` over `getByRole` for race-prone post-click queries** (line 389). After checking the row checkbox, React needs a re-render cycle to propagate `selectedIds` to `BulkActionsBar`. `findByRole` has an implicit `waitFor`, so the test doesn't need an explicit `await waitFor(() => screen.getByRole(...))` wrapper. Cleaner.
- **DRF server-error â†’ `inviteForm.setError()` propagation is end-to-end tested** (line 518-541). The mock rejects with `{ response: { data: { email: ['A user with this email already exists.'] } } }` â€” the exact shape returned by DRF field-validation errors â€” and the test asserts the message renders under the email field. This catches a regression where the error pathway breaks and falls back to the generic toast-only error.
- **Both directions of the Zod validation test** (line 504-516):
  ```ts
  await waitFor(() => expect(screen.getByText(/email is required/i)).toBeInTheDocument());
  expect(svc.createInvitation).not.toHaveBeenCalled();
  ```
  The negative assertion (`not.toHaveBeenCalled`) is the load-bearing one â€” without it, "validation message shows" + "API still fires" would silently pass.
- **Empty fixtures match the production schema.** `INVITATION_PENDING.status: 'pending'` (lowercase, matching DRF enum), `expires_at` as ISO string, `invited_by` as the inviter's email. No fabricated fields, no wrong types.
- **`vi.mock('react-router-dom', ...)` with `importOriginal`** (line 22-25) preserves all the other react-router exports (MemoryRouter, etc.) while replacing only `useNavigate`. The wrong way to do this â€” naked `vi.mock('react-router-dom', () => ({ useNavigate: ... }))` â€” would break MemoryRouter inside `renderPage()`. Done right here.
- **No `any` types in the test file.** Service mock cast at line 109-118 uses an explicit struct type with `ReturnType<typeof vi.fn>` per field, not a wildcard.
- **Tests-the-UX-not-the-DOM choices.** Edit modal panel asserted via the "Edit Teacher" heading (line 242) rather than `role="dialog"` (because it's a plain `<div>`, not Headless UI). If the implementation later swaps to a real `Dialog`, the test still passes because it asserts on visible UX, not implementation detail.
- **Production component file exists.** Import `import { TeachersPage } from './TeachersPage'` resolves to `frontend/src/pages/admin/TeachersPage.tsx`. âś…
- **Test counts in request note match the file.** Author claimed 23 tests, hand-count confirms 23. Author's "0 regressions vs 774 prior â†’ 797/797 passed" arithmetic is consistent (774 + 23 = 797).
