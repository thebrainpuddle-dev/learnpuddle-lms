## Major Issues

### M1 — Fake-pass guards in AttendancePage tests
**File:** `pages/student/AttendancePage.test.tsx:280-298, 310-323`

Both calendar nav tests filter buttons by CSS class + svg presence + empty text:
```ts
const navButtons = allButtons.filter(
  (btn) => btn.querySelector('svg') && !btn.textContent?.trim() && btn.className.includes('rounded-lg'),
);
if (navButtons.length >= 1) {
  await user.click(navButtons[0]);
  expect(...).toBeInTheDocument();
} else {
  expect(true).toBe(true);   // ← silently passes if filter is wrong
}
```

This is the classic **fake-pass anti-pattern**: if the Tailwind classes change, if the svg structure changes, if a sibling button gains an empty-text icon — the filter returns `[]` and the test falsely passes. The comment "guard: test infra can't find them" admits the test no longer verifies the behavior it claims.

**Fix:** Replace the brittle DOM-class filter with deterministic selectors:
- Add `aria-label="Previous month"` and `aria-label="Next month"` to the buttons in the component, and query by `getByRole('button', { name: /previous month/i })`.
- OR add `data-testid="cal-prev"` / `data-testid="cal-next"` and use `getByTestId(...)`.
- Then drop the if/else and assert unconditionally — if the selector fails, the test should fail loudly.

These two tests are the only blocker for approval.
