# Review: FE-031 — clearAllMocks → resetAllMocks sweep + ESLint guard + OutlineEditor contract

## Verdict: APPROVE

## Summary

Three small, well-scoped follow-ups from FE-028/029/030 advisory feedback: (a) a CONTRACT comment on `OutlineEditor.tsx`'s two `useMemo(validateOutline)` hooks tying them to TASK-062-L8's delta assertion; (b) an ESLint rule banning `vi.clearAllMocks()` in favor of `vi.resetAllMocks()`, co-located with the `useFakeTimers` rule in a single config object to dodge ESLint v9 flat-config override semantics; (c) a global sweep replacing 53 occurrences across 29 test files. tsc clean, 544/544 vitest pass, ESLint baseline unchanged. `grep` confirms zero remaining `vi.clearAllMocks()` calls.

## Critical Issues

None.

## Major Issues

None.

## Minor Issues

1. **ESLint rule selector matches `vi.clearAllMocks` only — not `globalThis.vi.clearAllMocks` or aliased `const v = vi; v.clearAllMocks()`**.
   The selector `CallExpression[callee.object.name='vi'][callee.property.name='clearAllMocks']` covers the canonical case used everywhere in the codebase today. An alias or namespace re-export would slip through. Acceptable risk — flag for awareness; don't expand the rule until it bites.

2. **Test count regression noted but not investigated** (cover note line 120: "544 vs 557 baseline").
   The frontend-engineer asserts no test failed and that the count drop predates FE-031. Worth a one-time `git log --diff-filter=D -- '*.test.tsx' '*.test.ts'` to confirm tests were intentionally deleted (e.g., during a feature rip-out) rather than accidentally lost. Won't block FE-031.

3. **CONTRACT comment couples the implementation to the test's exact upper bound (delta ≤ 2)**.
   Pros: future devs know to update the test if they add a third memo. Cons: invites lazy bumping of the upper bound rather than questioning whether the third memo is necessary. Consider phrasing the comment to add "If you need to add a third, prefer `useMemo` consolidation rather than a third call." Minor nit.

## Positive Observations

- **Co-locating both restrictions in one `no-restricted-syntax` array** is the right move for ESLint v9 flat config — separate config objects with different `files` selectors would have caused the second to override the first's restriction list. The header comment explicitly calls this out, which is excellent operator-friendly documentation.
- **The error message is teachable, not punitive**: it explains *why* `resetAllMocks()` is preferred, not just *what* to use. Reduces tribal-knowledge debt.
- **53 replacements with safety rationale**: the cover note documents that every replaced call was followed by mock re-establishment in the same `beforeEach`, so wiping implementations is safe. That's the precise correctness argument needed.
- **YAGNI discipline**: explicit decision to *not* refactor `ManualReminderType` to `Exclude<ReminderType, 'COURSE_DEADLINE'>` because auto-derivation could silently include future enum members. That's exactly the right call for a security-adjacent UI surface.
- **OutlineEditor CONTRACT comment** is the kind of cross-file constraint comment that prevents future refactors from breaking subtle test invariants. Worth replicating as a pattern for other "exact-count" assertions.
- **No regressions**: tsc + 544/544 vitest + zero `clearAllMocks` remaining + ESLint baseline preserved. Clean.

## Recommendation

**APPROVE** and merge. The minor issues are forward-looking suggestions, none blocking.
