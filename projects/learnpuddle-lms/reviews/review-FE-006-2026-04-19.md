## Major Issues

### M1. Hard-coded `learnpuddle.com` in `buildSpUrls` — breaks staging/dev

**File:** `frontend/src/pages/admin/SettingsPage.tsx:1062-1070`

```ts
function buildSpUrls(subdomain: string, spEntityId: string) {
  const base = `https://${subdomain}.learnpuddle.com`;
  return {
    entityId: spEntityId || `saml-sp:${subdomain}`,
    acsUrl: `${base}/api/v1/auth/saml/${subdomain}/acs/`,
    slsUrl: `${base}/api/v1/auth/saml/${subdomain}/sls/`,
    metadataUrl: `${base}/api/v1/auth/saml/${subdomain}/metadata/`,
  };
}
```

The platform domain is env-driven everywhere else in the app
(`frontend/.env.development: REACT_APP_PLATFORM_DOMAIN=localhost:3000`,
`frontend/.env.production: REACT_APP_PLATFORM_DOMAIN=learnpuddle.com`),
and we already have `getPlatformDomain()` in
`frontend/src/utils/hostRouting.ts:16`.

Hardcoding the prod domain means:
- **Staging** admins copy an ACS URL that points at production.
- **Dev** admins get nonsense URLs like `https://demo.learnpuddle.com/...`
  when the backend is actually running on `demo.localhost:8000`.
- An admin onboarding a real IdP from a non-prod environment will register
  the wrong ACS URL in their IdP — silent breakage.

**Required fix:** use the existing helper.

```ts
import { getPlatformDomain } from '../../utils/hostRouting';

function buildSpUrls(subdomain: string, spEntityId: string) {
  const platformDomain = getPlatformDomain() || 'learnpuddle.com';
  const scheme = platformDomain.includes('localhost') ? 'http' : 'https';
  const base = `${scheme}://${subdomain}.${platformDomain}`;
  return {
    entityId: spEntityId || `saml-sp:${subdomain}`,
    acsUrl: `${base}/api/v1/auth/saml/${subdomain}/acs/`,
    slsUrl: `${base}/api/v1/auth/saml/${subdomain}/sls/`,
    metadataUrl: `${base}/api/v1/auth/saml/${subdomain}/metadata/`,
  };
}
```

Add a test that verifies the URLs change when `REACT_APP_PLATFORM_DOMAIN`
is overridden (same pattern as `hostRouting.test.ts`).

## Minor Issues

### m1. Single-cert field hides a backend capability

**File:** `SettingsPage.tsx:176, 1113, 1154`

```ts
idp_x509_cert: z.string(), // single cert for UI; sent as idp_x509_certs[0] on submit
...
idp_x509_cert: samlConfig.idp_x509_certs?.[0] ?? '',
...
idp_x509_certs: data.idp_x509_cert ? [data.idp_x509_cert] : [],
```

Backend accepts `idp_x509_certs: string[]` and `verify_and_parse_response`
iterates them (`saml_service.py:276-292`) — this is how IdP cert rotation
works (both old and new cert accepted for the overlap window).

Today's UI collapses the array to one entry, so an admin rotating their
IdP cert has no way to keep the old cert around through the transition.
Either:
- (preferred) let them paste multiple certs (textarea with PEM concat or
  a "+ add certificate" button), OR
- call out in the help text that metadata-paste preserves all certs from
  the XML, and manual entry supports only one.

Not a blocker — metadata paste preserves multiple certs because the payload
goes through backend parsing. But if an admin uses manual fallback, they
lose rotation support.

### m2. Metadata XML re-posted on every save

**File:** `SettingsPage.tsx:1150`

Every save sends the full `idp_metadata_xml` back to the backend in the
PATCH payload (plus the parsed-out fields). Depending on backend behavior:
- If the backend re-parses `idp_metadata_xml` whenever it's non-empty, then
  any manual edits the admin made to `idp_entity_id` / `idp_sso_url` etc.
  will be silently overwritten on save. Admins "correcting" parsed values
  will find their corrections reverted.
- If the backend ignores the XML when URLs are already set, this is just
  wasted bandwidth.

Please verify backend behavior here. If (a), either clear
`idp_metadata_xml` from the form after successful parse, or only send it
on the dedicated "Parse & Auto-fill" button (the flow already does that
via `handleParseMetadata`) and strip it from the generic submit payload.

### m3. Textarea label not associated to control

**File:** `SettingsPage.tsx:1249-1266`

```tsx
<label className="block text-sm font-medium text-gray-700 mb-1">
  IdP Metadata XML
  ...
</label>
<Controller
  control={form.control}
  name="idp_metadata_xml"
  render={({ field }) => (
    <textarea {...field} ... />
  )}
/>
```

No `htmlFor` / `id` binding. Screen readers won't announce "IdP Metadata XML"
when focus enters the textarea. Other fields in this file use the same
pattern — worth fixing consistently, but non-blocking for this review.

### m4. `retry: false` on the SAML query is good, but there's no error state

If `getSAMLConfig` returns 403 (feature disabled) or 500, the card renders
the form with its default values and silently. Users could save, and the
save would also 403. Add an inline banner for `isError` cases with a
clear "SAML is not enabled — contact support" message.

### m5. Type safety nit: `as SAMLDefaultRole` in defaultValues

**File:** `SettingsPage.tsx:1094`

```ts
default_role: 'TEACHER' as SAMLDefaultRole,
```

Unnecessary cast — the literal `'TEACHER'` is already assignable. Dropping
the cast tightens the type so a typo (e.g. `'TEACHERR'`) would surface at
compile time instead of slipping past. Low priority.

## Positive Observations

- **Feature gate placement is correct:** `{features.saml && <SAMLSSOCard />}`
  in the Security tab (line 1514) matches backend's `tenant.features['saml']`
  check in `_load_tenant_saml`. Layered defense.
- **RHF + Zod throughout** — no raw `useState` form fields. Validation
  messages surface via `fieldState.error.message`. Good pattern.
- **`retry: false` on the SAML and policy queries** — prevents retry
  storms on 403 from feature-gated endpoints. Small detail, big UX win.
- **SP URLs explicitly separated into a helper** (`buildSpUrls`) — easy to
  update once when M1 is fixed.
- **Copy buttons** use `navigator.clipboard.writeText` with a toast
  confirmation. Good affordance for admins pasting URLs into their IdP.
- **Conditional rendering of IdP fields** (`{enabledValue && ...}`) keeps
  the disabled state visually distinct. Nice touch.
- **Azure AD / WS-Federation claim URIs pre-filled** as defaults —
  saves ~70% of onboarding friction for the most common IdPs.
- **Error extraction chain on mutation** is verbose but type-safe
  (line 1133-1142). Better than `(err as any)...`.
- **TS strict passes and all 246 tests green** per the request.
