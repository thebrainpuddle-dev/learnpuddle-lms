## Minor Issues

1. **Switch index addressing is fragile** (`RemindersPage.test.tsx:234-235`).

   ```ts
   const switches = screen.getAllByRole('switch');
   const overdueSwitch = switches[5]; // sixth rule, index 5
   ```

   If the default rules list is ever reordered or extended (e.g. a 7th rule gets prepended in the source), this test silently asserts on the wrong row. Two non-blocking improvements:

   - Anchor on the rule's title text and walk to its switch with `within(...)`, or
   - Add an `aria-label` like "Toggle Assignment overdue notification" on the switch and target by name.

   Both require either a test-only DOM tweak or a one-line component change. Defer-able; not a blocker.

2. **`expect(screen.getByText('5')).toBeInTheDocument()` for recipient_count** (line 382).

   Asserting on the bare digit `'5'` is brittle if any other small number renders elsewhere (e.g. a sent_count badge of 5 in the campaigns fixture). It works today because the preview panel is the only "5" on screen at that moment, but a tighter assertion would be `screen.getByText(/5\s+recipient/i)` or `within(previewPanel).getByText('5')`. Nit.

3. **`adminRemindersService.send` mock returns `{ sent: 5, failed: 0 }`** (line 159) but no test asserts on the resolved value's effect on the UI beyond "Reminders sent!" toast. If the page ever surfaces "5 sent, 0 failed" in the toast, this fixture is ready, but right now it's an unused field. Cosmetic.

4. **Real-timer debounce strategy is justified inline** (lines 428-432):

   ```ts
   // The teacher search has a 300 ms debounce ... we use real timers throughout
   // these tests — userEvent key-by-key typing takes ~50ms/char, so by the time
   // we `waitFor` the dropdown the debounce has already fired naturally
   ```

   Comment is clear but the `{ timeout: 3000 }` on the two `waitFor` calls effectively budgets ~10× the debounce window. That's a flake-resistant safety margin, not a worry, but on a busy CI runner those two tests could occasionally slow the suite. Consider: if CI runtime ever climbs, these are the candidates to revisit with a `vi.useFakeTimers({ shouldAdvanceTime: true })` config that lets RTL polling work. For now: accept.

5. **`fireEvent.keyDown(input, { key: 'Enter' })` and `{ key: 'Escape' }`** (lines 260, 273) bypass `userEvent` for keyboard events.

   `userEvent.keyboard('{Enter}')` would be more idiomatic and also fires the proper key-up sequence. `fireEvent.keyDown` works because the component handles `onKeyDown`, so functionally equivalent. Style nit.

## Positive Observations

- **Module-level `vi.mock()` for every external service** — no per-test setup churn, no leaked mock state between tests. `vi.resetAllMocks()` in `beforeEach` keeps each test hermetic.
- **Per-test fresh `QueryClient` with `retry: false`** — TanStack Query won't burn 3× retries on expected error-toast tests, keeping individual test runtime tight.
- **Right disambiguation strategy for ambiguous labels.** "Manual" appears in both the filter button and the HistoryRow badge; "History" in both the tab button and section heading. The tests handle this via `getAllByText().length >= N` (line 517) and unique adjacent-text assertions (the "log of all sent reminders" string at line 197). This is the documented pattern in the request note and the right call — the alternative (asserting only on the first match) would silently flip-flop with React render-order changes.
- **`disabled` assertion over toast assertion for the ASSIGNMENT_DUE-without-selection case** (line 332). The component's actual UX is `disabled={!isPayloadValid}` — testing that exactly is more accurate than asserting an error toast that the component doesn't fire. Tests-the-behavior, not tests-the-test-author's-mental-model.
- **Negative assertion + positive assertion in the schedule guard test** (lines 419, 421):
  ```ts
  await waitFor(() => expect(screen.getByText('Schedule required')).toBeInTheDocument());
  expect(svc.send).not.toHaveBeenCalled();
  ```
  Both directions covered. Catches "the toast appears AND the API also fires" regressions.
- **Empty-state and loading-state tests use the recommended pattern** (`mockReturnValue(new Promise(() => {}))` for the loading case at line 495). The `never resolve` trick keeps the loading state visible without timer mocking.
- **Subject-search filter test is two-sided** (lines 543-544): asserts the matching campaign IS visible AND the non-matching campaign is NOT visible. Catches both "filter does nothing" and "filter hides everything" regressions.
- **No `any` types in the test file.** (One occurrence of the word "any" in a comment, line 432; zero `: any` annotations.)
- **Fixtures are minimal and explicit** — `mockTeachers`, `mockAssignments`, `mockCampaigns` each contain just the fields the page reads, with type annotations from the actual service contracts (`User`, `ReportAssignment`, `ReminderCampaign`). No magical "complete fixture" objects.
- **Production component file exists at the path the test imports.** Import resolves: `import { RemindersPage } from './RemindersPage'` → `frontend/src/pages/admin/RemindersPage.tsx`.
