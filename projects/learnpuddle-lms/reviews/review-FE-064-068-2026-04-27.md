## Minor Issues

1. **MAICLibraryPage spinner queried by class.** `document.querySelector('.animate-spin')`
   (L141). Same coupling concern as the FE-059-063 bundle. Suggest a `data-testid="loading-spinner"`
   or `role="status"` aria-label as in TeacherStudyNotesPage. Non-blocking — the rest of the page's
   stable identifiers (titles, headings) are sound.

2. **MAICLibraryPage "New Classroom" picks `[0]` from `getAllByRole`.** `screen.getAllByRole('button',
   { name: /new classroom/i })[0]` (L221-222). The component renders the button twice (header +
   empty state), so the index is stable today but breaks if header layout changes. Lower-risk
   alternatives: scope to header `<div>` with a testid, or click the empty-state button explicitly
   (`mockListClassrooms` returning `[]` makes both visible — the test currently runs against the
   default empty list, so [0] = header button is reasonable).

3. **MAICLibraryPage relies on `getByTitle('Delete classroom')` / `getByTitle('Assign sections')`.**
   `title` attributes are stable handles for tests, but they're tooltip text — not accessible names
   for screen readers. The icon-only buttons should also carry `aria-label`. **Production a11y issue,
   not a test issue** — file as a separate FE ticket. Test workaround is acceptable.

4. **QuizPlayerPage mutable `mockStore` is non-obvious but correct.** The pattern works because
   `setLiveQuizStore` / `resetStore` mutate fields and `vi.clearAllMocks()` clears `vi.fn()` call
   history but does NOT replace the function instances. The `beforeEach` then re-binds
   `vi.fn()` instances on `start`, `setAnswer`, `next`, `prev`, `clear`, etc. (L206-213). This is
   correct but a future contributor could easily forget to re-bind a new function and end up with
   a stale, cleared mock. **Recommendation (non-blocking):** wrap the reset in a single
   `resetMockStore()` helper that includes both fields and fresh `vi.fn()` bindings.

5. **ReportBuilderDetailPage delete-schedule confirm button matched by literal `'Confirm'`** (L382).
   Works because the `ConfirmDialog` stub renders that exact text. If the test author later
   converts the stub to render the real dialog's button labels (e.g., "Delete"), the test will
   silently break. Tightening to a `data-testid` on the confirm button would future-proof.

6. **QuizPlayerPage Submit-Quiz happy path doesn't assert payload shape.** Test only checks
   `mockSubmitAttempt.toHaveBeenCalledTimes(1)` (L323) — it doesn't verify the answers map
   reaches the API. Reasonable scope for a smoke test, but a missed regression vector. Consider
   asserting on `submitAttempt.mock.calls[0][1]` includes the staged answers when `setAnswer`
   has been called.

## Positive Observations

- **ReportBuilderEditorPage / ReportBuilderDetailPage use `data-testid` everywhere.** The
  components have stable test hooks (`editor-name`, `editor-submit`, `editor-loading`,
  `detail-loading`, `detail-error`, `overview-panel`, `preview-panel`, `schedules-empty`,
  `new-schedule-btn`, `delete-schedule-${id}`, `schedule-toggle-${id}`). The admin team has done
  the right thing on the production side — the tests benefit from it. This is the gold-standard
  pattern.
- **Both editor modes (create + edit) tested as separate `describe` blocks** with their own
  `beforeEach` and the right "did we call create or update?" assertions — one of the most
  common bug-introducing branches. `expect(mockCreate).not.toHaveBeenCalled()` after an edit-mode
  submit (L292) is exactly the right negative assertion.
- **ReportBuilderDetailPage exercises real `@headlessui/react` TabGroup** rather than stubbing it
  (L304, L326, L335…). High-fidelity — actual mount/unmount of tab panels gets exercised, and
  the comment at L18-20 explains the design choice clearly.
- **Preview-panel `textContent` assertion** (L307-308) deliberately handles the "Click
  **Run now** to preview results." text-split-by-`<strong>` case — the right move and the
  comment explains why.
- **Schedule cadence/recipient assertions** verify the formatter output (`/at 8:00 UTC/`,
  `/2 recipients/`) — catches both cadence-rendering regressions and recipient-count bugs.
- **ChatbotBuilderPage validation-error path** asserts the toast message AND that `mockCreate`
  was NOT called (L191-193). Both halves are needed; many tests only assert the toast and miss
  the no-API-call branch.
- **ChatbotBuilderPage edit-mode hydration tests** verify both name input AND welcome message
  textarea load with their persisted values — separately, so one regression doesn't mask the
  other.
- **QuizPlayerPage covers the full quiz lifecycle**: bootstrap loading → question render →
  answer → submit → result screen → done navigation. The `setLiveQuizStore` helper makes the
  "reuse" path (already-running attempt) testable without faking the bootstrap async race.
- **"Previous disabled on first question" + "Submit Quiz on last question" + "Next on
  middle question"** trio (L278-301) covers the navigation state-machine cleanly.
- **MAICLibraryPage card-click navigation** asserts the route includes the dynamic `id`
  (`/teacher/ai-classroom/cls-42`), not just any path — verifies parameter wiring.
- **Per-test fresh QueryClient** with `gcTime: 0, retry: false` is consistent across all 5 files.
