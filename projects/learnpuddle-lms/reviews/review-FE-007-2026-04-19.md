# Review: FE-007 — Rubric Management + Grading UI

## Verdict: REQUEST_CHANGES

Strong component structure and a faithful mirror of the backend API. Three
issues prevent APPROVE:
1. Server-side pagination never surfaced — admins can't see rubrics past page 1.
2. Two overlapping search inputs (server + client) on the same page.
3. Claims in the review request that don't match the code (live total, Switch).

No critical issues. Once the pagination gap is fixed, this is ready to ship.

## Major Issues

### M1. Server-side pagination not surfaced — data loss beyond page 1

**File:** `frontend/src/pages/admin/RubricPage.tsx:513-518`

```tsx
const { data, isLoading } = useQuery({
  queryKey: ['rubrics', search],
  queryFn: () => adminRubricService.listRubrics({ search: search || undefined }),
});
const rubrics = data?.results ?? [];
```

`adminRubricService.listRubrics` returns a DRF-paginated
`{ count, next, previous, results }`, but the page never reads `data.count`
or `data.next`, never passes `page` to the service, and feeds `results`
directly into `<DataTable>`. TanStack Table then paginates client-side
within that first page only.

With the default DRF page size (typically 10–20), tenants with >20 rubrics
will silently lose access to the rest of the library. The list query's key
also doesn't include a page number, so even manually paginating via a URL
param wouldn't work.

**Required fix (options):**
- Add server-side pagination UI: wire `page` state into `useQuery`
  (`queryKey: ['rubrics', search, page]`), pass it into `listRubrics`, and
  render previous/next controls using `data.next` / `data.previous` /
  `data.count`.
- Or: if the product expects <100 rubrics per tenant and the backend is OK
  with it, pass `?page_size=500` (or similar) and keep client-side
  pagination — but verify the backend cap first. This is a product call.

### M2. Two competing filter inputs on the same page

**File:** `RubricPage.tsx:672-688` (server-side search) +
`RubricPage.tsx:722-727` (`DataTable filterColumn="title"`)

The page renders its own `<input type="search">` that drives the
`search` state → server-side search param, and `<DataTable>` *also*
renders its own built-in filter box (client-side TanStack filter on the
`title` column). End users see two boxes labeled similarly, with different
behavior:
- Top input → server query, debounced (well, actually not debounced — see m1 below).
- DataTable input → instant client-side filter on already-loaded rows.

Pick one. Recommended: keep the server-side search for correctness across
pagination, drop `filterColumn` / `filterPlaceholder` from `<DataTable>`.

### M3. Review request claims features that aren't implemented

**File:** `RubricPage.tsx`

The review request says:

> - Total points auto-computed and displayed live in the modal footer as
>   criteria/levels change

The modal footer (lines 485-493) contains only Cancel and Save buttons.
No running total is computed from `useWatch` on `criteria`. If this was
dropped intentionally, update the review request; otherwise add it (it's
a 5-line `useWatch(control, 'criteria')` + `reduce` sum).

> - `is_active` toggled by a Switch (shadcn/ui pattern, rendered as Badge
>   in table)

The modal uses a native `<input type="checkbox">` (lines 426-436), not
a shadcn `<Switch>`. The Badge-in-table rendering is correct (lines
597-605). Either wire up the Switch component or update the claim.

Call this a "major" issue because the checklist is our source of truth
for what shipped; mismatches erode trust in future reviews.

## Minor Issues

### m1. Search input fires on every keystroke with no debounce

**File:** `RubricPage.tsx:672-678`

```tsx
<input
  type="search"
  value={search}
  onChange={(e) => setSearch(e.target.value)}
  ...
/>
```

Every character in the search box changes `search`, which changes the
`useQuery` key, which triggers a new GET. For a tenant with many rubrics
this is noisy at best, rate-limited at worst. Debounce with a 250–300ms
delay (e.g. `useDeferredValue` or a small `useDebouncedValue` hook we
probably already have).

### m2. Generic error toast swallows backend validation errors

**File:** `RubricPage.tsx:367-369`

```tsx
} catch {
  toast.error('Failed to save', 'Please check your inputs and try again.');
}
```

When the backend returns 400 with a structured validation payload (e.g.
`{ criteria: [{ levels: [{ points: ['Must be <= max_points'] }] }] }`),
users see only "Please check your inputs and try again." — with no
indication of which field is broken. Zod covers most of this client-side,
but server-authored rules (unique title per tenant, max_points sum
constraints if they exist) leak through.

Extract the error detail the way `SettingsPage.tsx` does (see
`SAMLSSOCard.onError`) and surface the first field error.

### m3. Modal is not keyboard-accessible

**File:** `RubricPage.tsx:372-494` (RubricModal) and 81-111 (ConfirmDialog)

- No `Escape` key handler — backdrop click works, keyboard users are stuck.
- No `role="dialog"` / `aria-modal="true"` / `aria-labelledby` on the
  modal container. Screen readers treat it as ordinary content.
- Focus trap absent — Tab escapes the modal into the page behind it.

If the app has a reusable `<Dialog>` primitive (shadcn/radix-ui), this
should use it. If not, add at least the Escape handler and
`role="dialog"` — they're 5 lines of boilerplate.

### m4. Callback-prop save flow is awkward

**File:** `RubricPage.tsx:338-370` + `RubricPage.tsx:555-557`

The modal does the create/update mutation itself (`try/catch` inside
`onSubmit`), then calls `onSaved()` to trigger query invalidation in the
parent. Every other mutation on this page uses `useMutation` properly.
Consolidating would let you:
- Reuse the mutation's `isPending` state instead of `isSubmitting`.
- Put `onSuccess: () => invalidateQueries` in one place.
- Surface errors via `onError` rather than a try/catch.

Cosmetic — no functional bug.

### m5. `deleteTarget?.title` briefly shows "undefined" during close

**File:** `RubricPage.tsx:744`

```tsx
title={`Delete "${deleteTarget?.title}"?`}
```

When the delete succeeds and we call `setDeleteTarget(null)`, there's a
render cycle where `isOpen={!!null} === false` but the title was built
from `deleteTarget?.title` → `"Delete "undefined"?"`. The dialog is
hidden by this point (`if (!isOpen) return null`), so no visible impact.
Consider using `row.original.title` snapshot at click time, or guard the
render inside the dialog.

### m6. `is_active` checkbox register pulls in string not boolean

**File:** `RubricPage.tsx:428`

`{...register('is_active')}` on a checkbox — RHF returns `true/false`
for `type="checkbox"`, so the Zod schema's `.boolean()` works fine. But
since the input doesn't have `valueAsBoolean` explicitly set, the
payload's `is_active` might serialize as `"true"` string when read via
`getValues()` in some edge cases. Current tests pass, so this is
speculative — but worth a targeted test.

### m7. `feature: null` on sidebar entry

**File:** `frontend/src/components/layout/AdminSidebar.tsx:78`

`feature: null` means the Rubrics nav item is always visible, regardless
of tenant plan/features. If rubrics should be plan-gated (Pro/Enterprise
only), this will silently expose the feature to FREE tenants. If it's
meant to be universal, add a comment to that effect. Product call.

## Positive Observations

- **Zod schemas are precise and layered** — `RubricLevelSchema` →
  `RubricCriterionSchema` → `RubricSchema`, matching the nested domain
  model exactly. `.coerce.number()` on `points` / `max_points` correctly
  handles `<input type="number">` string returns.
- **`useFieldArray` nested inside a parent `useFieldArray`** is
  notoriously tricky to wire — `CriterionCard` gets `control` from parent
  and uses the `criteria.${index}.levels` path correctly. No state sync
  bugs.
- **Clone + invalidate flow** uses `useMutation` + `onSuccess` properly —
  no manual refetch, no full page reload. Clean.
- **Delete uses `ConfirmDialog` with variant="danger"`** — no raw
  `window.confirm`, no `alert()`. Matches the rest of the admin shell.
- **Empty state is thoughtful**: distinguishes "no rubrics yet"
  (call-to-action to create first) from "no rubrics match search"
  (no CTA, just an explanation).
- **`react-query` key includes `search`** — server-side search results
  are correctly cache-keyed.
- **Code splitting** via `React.lazy` + `<RoutePage>` is wired up in
  `App.tsx:133-134, 453`. Good.
- **All interactive elements have `cursor-pointer`** and `focus:ring-*`
  for keyboard navigation — consistent with the rest of the admin UI.
- **Heroicons throughout**, no emoji, consistent with sidebar's
  `ClipboardDocumentCheckIcon`.
- **TS strict + 246/246 tests** per the request. (Didn't re-run locally;
  trusting the assertion.)
