# Review: FE-078 / FE-091 / FE-092 — REQUEST_CHANGES Fixes

## Verdict: APPROVE

## Summary

Both REQUEST_CHANGES items from the 2026-04-29 batch are addressed cleanly.
The AttendancePage fake-pass guard is gone (replaced with `aria-label`-driven
`getByRole` queries that fail loudly), and the SecuritySettings unlink button —
which was previously a dead `<Button>` with no `onClick` — now has a real

unlink fix actually goes one strict step beyond what the prior review demanded:
the previous handler was missing entirely (button rendered but did nothing),
so the new mutation is a real bugfix on top of the test-coverage fix.

## Minor Issues

### m1 — SSO provider id fixture diverges from real backend contract

`SecuritySettings.test.tsx` fixtures use `provider: 'google'` /
`{ id: 'google' }`, but `backend/apps/users/sso_views.py:37-40` returns
`{ 'id': 'google-oauth2', ... }`, and the real `UserSocialAuth.objects.get(...,
provider=provider)` lookup expects the social-django backend name
(`google-oauth2`).

The new test still passes because it asserts the same string the fixture
provides, but the fixture is no longer representative of the real API contract,
so a future regression where the FE forgets to forward `provider.id` verbatim
(e.g. someone hand-codes `'google'`) would not be caught. Cheap fix when
revisited:

```ts
const SSO_PROVIDERS_WITH_GOOGLE = {
  providers: [{ id: 'google-oauth2', name: 'Google', icon: 'google', auth_url: '...' }],
};
const SSO_STATUS_LINKED = {
  has_password: true,
  linked_providers: [{ provider: 'google-oauth2', uid: '12345' }],
  can_unlink: true,
};
```

Then update the assertion to `expect.objectContaining({ provider: 'google-oauth2' })`.
This isn't blocking — the production component code (`provider.id`) is
correct — but the fixture should track reality.

### m2 — Calendar tests are still time-dependent

`AttendancePage.test.tsx` tests 7, 8, 9, 14, 15 all use `new Date()` to compute
the current month string and then assert against the calendar header. The mock
data is hardcoded to `'2026-04'`. Today is 2026-04-30 so they pass; on
2026-05-01 the component will request `/v1/student/attendance/?month=2026-05`
and the test will still seed it with the April mock, so the assertions about
"previous month = March" become wrong (previous month from May is April).
Not introduced by this fix — pre-existing in the suite — but worth a
`vi.useFakeTimers({ shouldAdvanceTime: true }); vi.setSystemTime(new
Date('2026-04-15T00:00:00Z'))` in a `beforeAll` next time the file is touched,
so the tests don't silently start failing in May.

### m3 — `getAllByText(...).length >= 1` softness in test 10

The legend-labels test uses `getAllByText('Present').length >= 1` because the
status label appears in both the legend row and the per-day tooltip overlay.
The looseness is justified, but the assertion would also pass if the legend
row disappears entirely and only the overlay remains. A tighter version that
still works around the overlay duplication:

```ts
const legend = container.querySelector('.calendar-legend, [data-testid="calendar-legend"]');
expect(within(legend!).getByText('Present')).toBeInTheDocument();
// ... etc.
```

…or scope the legend with a `data-testid` on the legend `<div>`. Not blocking.

## Positive Observations

### FE-078 — fake-pass guard removed, fix landed in component too

- The diff is exactly what the prior review prescribed: `aria-label="Previous
  month"` / `aria-label="Next month"` added to the icon-only buttons in
  `AttendancePage.tsx:166-180`, and the tests now use
  `screen.getByRole('button', { name: /previous month/i })`.
- The accessibility win is real — icon-only buttons without `aria-label` were
  unannounced for screen readers; this fixes that as a side-effect.
- Tests 8 and 9 now fail loudly if the selector misses (no
  `expect(true).toBe(true)` fallback anywhere). I diff'd the test file to
  confirm — the `if (navButtons.length >= N) { ... } else { expect(true)... }`
  pattern is gone from both call sites.
- The "bonus" fixes for tests 7 and 10 (the `multiple elements found`
  failures) are correctly explained in the request and the rationale matches
  what the DOM actually renders (h3 in both AttendanceCard and calendar
  header; legend label duplicated in tooltip overlays).

### FE-092 — Unlink button now actually works + tested

- The previous component had `<Button variant="outline" size="sm">Unlink</Button>`
  with **no** `onClick` — it was a dead button. The fix adds a real
  `useMutation` that POSTs `/users/auth/sso/unlink/`, hooks up `onClick`,
  passes `loading={…isPending}` for the disabled-during-flight UX, and
  invalidates `['sso-status']` on success so the linked-providers row
  re-renders.
- Imports were already present in the file (`useMutation`,
  `useQueryClient`) — no churn.
- The new test follows the file's existing mock pattern (`mockedApiPost`) and
  uses `findByRole` correctly to wait for the linked state to render before
  clicking.
- `expect.objectContaining({ provider: 'google' })` is the right shape — does
  not over-constrain on extra body fields and is robust to the future
  addition of an idempotency key, csrf token, etc.
