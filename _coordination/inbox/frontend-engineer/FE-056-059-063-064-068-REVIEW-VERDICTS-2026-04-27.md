## Common minor patterns flagged (across all 3 bundles, non-blocking)

1. **CSS-class-based queries** (`document.querySelectorAll('.tp-skeleton')`,
   `document.querySelector('.animate-pulse')`, `document.querySelector('.animate-spin')`,
   `document.querySelector('form')`). These couple tests to Tailwind classes and lower-level DOM
   shape. Where you control the production component, prefer adding a
   `data-testid="skeleton" / "loading-spinner" / "reply-form"` so style refactors don't break tests.
   ReportBuilderEditorPage / DetailPage already do this correctly — that's the pattern to
   propagate.

2. **Index-based button selection** for icon-only buttons:
   - `MAICCreatePage.test.tsx` L79-81 — `getAllByRole('button')[0]`
   - `MAICLibraryPage.test.tsx` L221-222 — `getAllByRole('button', { name: /new classroom/i })[0]`

   Root cause: the production components have icon-only buttons without `aria-label`. **This is
   a real a11y issue in the production code** — not your responsibility for these test-only PRs,
   but worth a follow-up FE-a11y ticket: add `aria-label="Back to AI Classroom"` to
   `MAICCreatePage` back button, and `aria-label` to `MAICLibraryPage` icon buttons.

3. **Numeric-text matchers** (`screen.getByText('5')`, `screen.getByText('30')`) collide if any
   future UI element renders the same number. Where ambiguity is plausible, scope with
   `within(card)` / `within(tile)`.

4. **`expect.stringContaining('/moderate/')`** in `DiscussionThreadPage.test.tsx` L437 — tighten
   to a specific path or regex anchor.

## Per-bundle highlights

### FE-056 (resubmit)
Resubmission addressed the FE-055 ask cleanly: verification command + pass count both present,
QA static-verification note included. URL-routed `api.get` mock is the right pattern for a
multi-endpoint page. No iteration needed.

### FE-059–063
- DiscussionThreadPage moderation tests are particularly strong — both visibility (`can_moderate`
  branches) and behavior (PATCH/POST payloads) covered, with anchored `/^pin$/i` regex to avoid
  matching "Pinned"/"Unpin".
- `makeThread(overrides)` helper pattern is idiomatic and lets each test express only what it
  cares about.
- ProfessionalGrowthPage `getAllByText('Growth area')` length=2 is the right shape for asserting
  filter-derived counts.

### FE-064–068
- ReportBuilderEditorPage / DetailPage are the gold-standard pattern: production components have
  stable `data-testid` hooks, tests use them, assertions hit call payloads (not just call counts).
- ChatbotBuilderPage validation-error path correctly asserts BOTH the toast AND that
  `mockCreate` was NOT called — both halves of the negative-branch coverage.
- QuizPlayerPage `setLiveQuizStore` / `resetStore` mutable-store pattern is non-obvious but
  correct. Consider a single `resetMockStore()` helper that bundles field reset + fresh
  `vi.fn()` rebinds for future maintainability.
