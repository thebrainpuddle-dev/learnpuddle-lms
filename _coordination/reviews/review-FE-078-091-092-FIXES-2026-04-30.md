# Review: FE-078 + FE-091/092 — REQUEST_CHANGES Fixes

## Verdict: APPROVE

## Summary
Both REQUEST_CHANGES items from the 2026-04-29 batch are addressed cleanly. FE-078: the brittle `allButtons.filter(...)` + `if (...) { } else { expect(true).toBe(true) }` fake-pass pattern is gone, replaced with semantic `getByRole('button', { name: /previous month/i })` queries — and the component now exposes the matching `aria-label`s, which is also an a11y win. FE-092: the previously-orphan "Unlink" button now has an `onClick` that wires through to the `/users/auth/sso/unlink/` endpoint via a real `useMutation`, with a test that asserts the contract. As a bonus, two pre-existing flaky tests (Tests 7 and 10) were narrowed and now pass.

## Minor Issues

1. **`getAllByText(...).length >= 1` is weaker than the original intent.** Test 10 ("renders all four calendar legend labels") used to use `getByText`, then crashed on duplicate matches. Switching to `getAllByText(...).length >= 1` *does* fix the false failure, but `.length >= 1` is a weak assertion — it accepts 1 or 99. Consider `.length === <expected count>` if the legend really should render exactly N elements, or scope the query (`within(legendContainer).getByText(...)`) so a single match is unambiguous. Non-blocking.

2. **Unlink mutation does not surface API errors.** `SecuritySettings.tsx:251` adds `unlinkProviderMutation` but the snippet shown only invalidates `['sso-status']` on success. If the mutation has `onError` handling (toast / inline error), great; if not, a failure mode (404, 500) would silently leave the button stuck in `isPending=false` with no user-visible feedback. Worth a quick check that the existing error-handling path applies here. Non-blocking — could be a follow-up.

3. **Test names could be slightly more specific.** `'navigates to the previous month when the left-chevron button is clicked'` is fine, but with the new `aria-label="Previous month"`, the button is no longer "left-chevron" — it's just "Previous month". Minor cleanup. Non-blocking.

## Positive Observations

- **a11y improvement piggybacks on the test fix.** Adding `aria-label="Previous month"` / `aria-label="Next month"` makes the icon-only buttons accessible to screen readers. The right test fix *demanded* the right component fix.
- **Dropped the `if (navButtons.length >= N) { ... } else { expect(true).toBe(true) }` fake-pass.** This is exactly the anti-pattern that prompted the REQUEST_CHANGES — the new tests fail loudly if the selector misses. Confirmed via grep that `allButtons.filter` no longer appears in `AttendancePage.test.tsx`.
- **Real mutation, not a stub.** `unlinkProviderMutation` is a proper `useMutation` that hits the actual endpoint, invalidates the SSO status query, and is wired to the button's `loading` prop so it disables during the request. The test asserts `mockedApiPost` was called with `'/users/auth/sso/unlink/'` and `expect.objectContaining({ provider: 'google' })` — that's contract-level, not implementation-level.
- **Bonus pre-existing test fixes (Tests 7 and 10).** Disclosed transparently in the request, with the specific cause for each (h3 ambiguity, duplicate legend text). 21/23 → 23/23 net improvement.
- **Combined run clean.** 66/66 across both files, no regressions.
