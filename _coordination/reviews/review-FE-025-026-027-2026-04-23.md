## FE-025 — `ReminderPayload` discriminated union + assignment picker

### Verdict: APPROVE

### Findings

**Discriminated union soundness (`adminRemindersService.ts`)**

- `ReminderType` is now the exact 3-value literal union matching the backend
  `ChoiceField`. Good.
- `AssignmentDuePayload` vs `NonAssignmentPayload` uses the standard
  `Exclude<ReminderType, 'ASSIGNMENT_DUE'>` for the non-assignment branch.
  Correctly prohibits `assignment_id` on the non-assignment side via
  `assignment_id?: never`.
- `ReminderCampaign.reminder_type` upgraded to `ReminderType` — catches
  typos at consumer sites. Good.
- The compiler enforces the discriminant at call sites (verified by inspecting
  `ManualSendSection.tsx:60-73` and `ReportDrillDown.tsx:506-511`).

**Assignment picker UI (`ManualSendSection.tsx`)**

- Conditional render on `reminderType === 'ASSIGNMENT_DUE'` (L188-207). Label
  has a proper `htmlFor` linkage, a red-asterisk required indicator, and a
  placeholder "Select an assignment…" option with empty value. Accessibility
  looks correct.
- `useQuery({ enabled: reminderType === 'ASSIGNMENT_DUE' })` (L85) prevents
  the assignments fetch from firing when the picker is hidden. Good.
- `isPayloadValid` guard (L142) correctly disables Preview and Send buttons
  when ASSIGNMENT_DUE is selected without an assignment. `handleSend` also
  shows a toast on attempted send in that state — belt-and-suspenders, good.
- On type switch (L176-179), `assignmentId` is cleared, so dangling state
  can't survive a user toggling the dropdown.
- `onError: (error: unknown)` with explicit cast — passes strict mode, no
  `any` leaks.

**Minor, non-blocking observations**

- Local `ReminderType` alias at `ManualSendSection.tsx:20` is narrower than
  the service-layer `ReminderType`. It shadows the exported name. Recommend
  renaming to `ManualReminderType` or reusing the service type directly so
  a future "add COURSE_DEADLINE to manual composer" change doesn't need
  two edits. Non-blocking.
- `reminderPayload` is built eagerly on every render; fine here since it's
  cheap, but could become stale in a `useCallback` scenario. Current code
  is correct.

No blockers.

## FE-027 — ChatPanel clear-chat tests

### Verdict: APPROVE

### Findings

**Mocking strategy is appropriate**

- `maicDb.updateClassroomChat` — mocked to avoid IndexedDB setup in
  happy-dom. Spied on for assertion.
- `maicChatSession.{hydrateChatFromSession,persistChatToSession}` — mocked
  and asserted on the confirm path.
- `maicSSE.streamMAIC` — stubbed to prevent any real streaming during
  unrelated clear-chat assertions.
- `useSpeechInput` — stubbed as unsupported.
- Sub-components (`PromptInput`, `ConversationContainer`, `StreamMarkdown`,
  `AgentAvatar`, `ChainOfThought`, `CodeBlock`) stubbed to minimum — correct
  scope.

**Coverage vs ChatPanel behaviour**

Verified that `ChatPanel.tsx` actually has:
- Toolbar button at L570-583: `aria-label="Clear chat"` + `<span>Clear chat</span>`.
- `ConfirmDialog` at L791-798 with `confirmLabel="Clear chat"` and
  `cancelLabel="Keep messages"`.

The test uses two disambiguation strategies cleanly:
1. `getByRole('button', { name: /clear chat/i })` for the toolbar button
   (accessible name comes from aria-label).
2. `getByText('Clear chat', { selector: 'button' })` for the dialog's
   confirm button — this works because RTL's `getByText` matches the
   narrowest element whose direct text is "Clear chat"; the toolbar button's
   text is inside a `<span>`, so `getByText` returns the `<span>`, but the
   `{ selector: 'button' }` filter excludes the span, leaving only the
   dialog button whose direct text is "Clear chat". Clever and correct.

**Test-pattern note**

- The `mockClear()` after `waitFor(...)` pattern to distinguish mount-time
  effects from user-interaction side-effects is the correct approach.
  Documented inline — good.

**Minor (non-blocking)**

- The last test (`handles a no-op gracefully when "Clear chat" is called on
  an already-empty store`) is slightly misleading — it actually verifies
  that the button *isn't rendered* at all, not that an invoked clear is a
  no-op. That's defensive but weaker than what the title suggests. The
  current assertion is correct; consider either renaming the test or
  directly invoking the clear action. Non-blocking.

No blockers.
