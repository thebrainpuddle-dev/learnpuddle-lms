# Review: FE-030 — Non-blocking follow-ups from FE-025 / FE-027 / FE-028 reviews

## Verdict: APPROVE

## Summary
Three small, focused follow-ups landing the non-blocking suggestions from earlier review verdicts:
(a) `ReminderType` → `ManualReminderType` rename to stop shadowing the service export,
(b) ChatPanel test name clarified to match the actual assertion, and
(c) aiCourseGenerator `validateOutline` upper-bound assertion with a single-character type so
the bound is meaningful. All three are clean, minimal, and well-reasoned.

## Critical Issues
None.

## Major Issues
None.

## Minor Issues

1. **FE-030c upper bound of 2 is load-bearing but undocumented in the component.**
   The bound "delta ≤ 2" depends on `OutlineEditor` having exactly two `useMemo(validateOutline, …)`
   hooks (verified at `pages/admin/ai-course-generator/components/OutlineEditor.tsx:279,282`). If
   someone adds a third `validateOutline` memoization in the future (say, for an extra view), this
   test will go red with no obvious connection to the change. Consider a brief code comment at the
   two `useMemo` sites — e.g. `// aiCourseGenerator.test.tsx L704 asserts exactly two of these` —
   so the implicit contract is discoverable from the production side. Non-blocking.

2. **FE-030c relies on `userEvent.type` producing exactly one outline state change.**
   For single-character input this is true with `@testing-library/user-event@14`, but userEvent
   internally fires `keyDown` + `input` + `keyUp`. If the `<input onChange>` path were ever
   debounced at the component level (it isn't today — only `debouncedOutline` is), the assertion
   could become fragile. Current behavior is correct; just flagging for future contributors.

3. **FE-030a — `ManualReminderType` stays narrower than the service `ReminderType` by design**
   (no `COURSE_DEADLINE`). The inline comment (line 13-15) explains why. Long-term, a cleaner
   pattern would be `type ManualReminderType = Exclude<ReminderType, 'COURSE_DEADLINE'>` so the
   two stay in sync automatically — but that's a refactor, not a fix, and out of scope.

## Positive Observations

- **FE-030a** correctly addresses the "shadows service export" concern from the FE-025 review. The
  comment block (lines 14-16) documents the intentional narrowing so the next reader doesn't try
  to unify the two types.
- **FE-030b** — test name now matches body. The old name ("handles a no-op gracefully when …
  called on an already-empty store") implied the handler was invoked directly; the new name
  ("button is not rendered and no side-effects fire when the store is already empty") describes
  the actual guard-clause assertion. Textbook feedback-driven clarification.
- **FE-030c upper-bound rationale is sound.** Verified `OutlineEditor.tsx` does have exactly two
  `useMemo(validateOutline, ...)` — one on `outline` (L279), one on `debouncedOutline` (L282) —
  so `delta ≤ 2` is correct. The single-character `userEvent.type(titleInput, 'X')` is the key
  change that makes the bound meaningful; typing 13 characters as before made the bound trivially
  false. Good analysis.
- **No production code changes in FE-030a/b, FE-030c is test-only.** Zero risk of regression.
- **557/557 full suite** passes, `tsc --noEmit` clean.

Frontend-only, no API/security impact.
