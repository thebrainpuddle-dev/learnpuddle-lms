# Review: FE-028 ‚Äî Fix aiCourseGenerator stack overflow + CourseEditorPage hash-scroll

## Verdict: APPROVE

## Summary
Two surgical fixes ‚Äî one test-only (stack-overflow spy recursion in the TASK-062 L8 test) and one
production bug in `useCourseForm.ts` (hash fragment silently stripped during tab normalization).
Both root-cause analyses are correct, the fixes are minimal, and the new `CourseEditorPage` hash-scroll
test exercises the production fix end-to-end.

## Critical Issues
None.

## Major Issues
None.

## Minor Issues

1. **`useCourseForm.ts` still imports `useSearchParams`** (line 7 / line 29) even though the
   normalization effect no longer uses `setSearchParams`. `setActiveTab` (line 74) still uses it, so
   the import is not dead ‚Äî but the fact that both paths exist (`setSearchParams` for user-clicks,
   `navigate` for auto-normalization) is a subtle split that future readers will trip over. Not
   blocking; a one-line comment on `setActiveTab` ("hash-preservation not needed on user tab click ‚Äî
   see normalization effect at L84 for the hash-stripping pitfall") would pay for itself.

2. **`as any` casts in the new `CourseEditorPage` test** (lines 207, 221, 228, 229 of the diff) are
   understandable for the stubbed `api` responses but are exactly the kind of thing the ESLint
   `no-explicit-any` rule is there to catch. Consider `as Partial<AxiosResponse<unknown>>` or a
   small `mockCourseResponse()` helper. Non-blocking ‚Äî the rest of the file already uses `as any`
   the same way.

3. **Hash-scroll timer of 300 ms is implicit contract**. The production code waits 300 ms before
   calling `scrollIntoView` (line 83 of `CourseEditorPage.tsx`); the test polls with a 1 s
   `waitFor` timeout. Fine for now, but extracting the delay to a named const (`HASH_SCROLL_DELAY_MS`)
   would let the test import it rather than silently depending on it.

## Positive Observations

- **Root cause correctly diagnosed for Fix 1.** `vi.spyOn(obj, 'key')` mutates `obj.key` to be the
  spy; capturing the original reference BEFORE `spyOn` is the textbook fix. The inline comment
  (lines 658-661 of `aiCourseGenerator.test.tsx`) will save the next person hours.
- **Root cause correctly diagnosed for Fix 2.** `useSearchParams`'s `setSearchParams` internally
  calls `navigate("?" + newParams)`, which drops the hash ‚Äî verified by tracing react-router-dom
  source. Replacing with `navigate(\`?${params}${location.hash}\`, { replace: true })` is the
  minimal correct fix.
- **Dependency array correctly updated** in the effect (line 91): `location.hash` and `navigate`
  added, `setSearchParams` removed. Matches ESLint `react-hooks/exhaustive-deps`.
- **`setActiveTab` intentionally NOT changed.** The reasoning in the review request ("user
  intentionally navigating away from hash-anchored content") is sound ‚Äî tab clicks discard the
  anchor by design, preventing the scroll from firing again on tab switch.
- **Hash-scroll test plants a real DOM element and asserts the exact scroll args**
  (`{ behavior: 'smooth', block: 'center' }`). This is a behavior test, not an implementation test ‚Äî
  if the scroll implementation changes, only the args assertion needs updating.
- **557/557 full suite after FE-029**, `tsc --noEmit` clean.

## Scope & Tenant-isolation
Frontend-only, no API surface change, no security impact.
