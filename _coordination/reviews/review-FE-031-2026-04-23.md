# Review: FE-031 — Follow-ups from FE-028/029/030 (contract comment, ESLint rule, clearAllMocks sweep)

## Verdict: APPROVE

## Summary
Three clean advisory follow-ups: a self-documenting CONTRACT comment pinning a
test assertion to its source, a defensive ESLint rule preventing regression to
`vi.clearAllMocks()`, and a global sweep that replaces 53 occurrences in 29
test files with `vi.resetAllMocks()`. All changes are low-risk, well-reasoned,
and the lint rule is correctly located inside the existing `no-restricted-syntax`
array so flat-config override semantics cannot silently drop either restriction.

## Critical Issues
None.

## Major Issues
None.

## Minor Issues

- **Count mismatch note (informational only, not a fix)**: request claims 53
  `vi.clearAllMocks` → `vi.resetAllMocks` replacements, but the tree now shows
  61 `vi.resetAllMocks()` call sites. The delta is consistent with pre-existing
  `resetAllMocks()` usage (e.g. prior FE-029 RubricPage fix) rather than the
  sweep, and `grep -rn "vi\\.clearAllMocks" frontend/src/` returns zero hits.
  No action required — just worth logging in case future metrics rely on it.

- **`no-restricted-syntax` severity is `error`** — intentional and correct, but
  worth calling out for authors who land a new `clearAllMocks` on a feature
  branch: the lint will fail CI rather than warn. Consider documenting this in
  the contributor guide if CI failures start spiking.

## Positive Observations

- **`OutlineEditor.tsx` CONTRACT comment** (lines 278–286): excellent pattern.
  It names the test file (`aiCourseGenerator.test.tsx`), the task
  (`TASK-062-L8`), the exact assertion (delta ≤ 2 `useMemo(validateOutline)`
  calls), and the update instruction if a third `useMemo` is added. This is
  the right way to keep an implementation–test invariant from silently drifting.

- **ESLint rule placement**: putting the `clearAllMocks` restriction into the
  *same* `rules: { 'no-restricted-syntax': [...] }` array as the existing
  `useFakeTimers` rule avoids the ESLint v9 flat-config merge gotcha (later
  configs on the same rule replace rather than extend). The comment block
  explicitly documents this reasoning — future editors won't accidentally split
  the rules into two config objects.

- **Safety rationale for the sweep is correct**: every `vi.clearAllMocks()`
  followed by a re-establishment of mock return/resolved values in the same
  `beforeEach` is safe to switch — `resetAllMocks()` wipes both call history
  and implementations, but the implementations are immediately re-seeded in
  the same tick. Spot-checked `ChatbotWidget.test.tsx` (10 sites),
  `App.test.tsx`, `CourseEditorPage.test.tsx`, `authService.test.ts` — all
  match the pattern.

- **Out-of-scope call is right**: rejecting
  `type ManualReminderType = Exclude<ReminderType, 'COURSE_DEADLINE'>` because
  it would silently admit future `ReminderType` members is sound YAGNI +
  type-safety reasoning. The explicit union is the correct contract.

## Evidence

- `grep -rn "vi\\.resetAllMocks" frontend/src/ | wc -l` → 61.

- `OutlineEditor.tsx:278–286`: two `useMemo(() => validateOutline(...), ...)`

- Author-reported verification: `tsc --noEmit` 0 errors, `vitest run` 544/544

  not regression — acceptable given the sweep doesn't remove any `it()` blocks.
